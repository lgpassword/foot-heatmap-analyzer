# 算法与工程说明

本文档说明 `foot-heatmap-analyzer` 当前已经实现的主要算法、数据流和工程边界。项目定位是足底压力数据分析原型，用于研究、教学、设备接入验证和开源协作，不提供医疗诊断结论。

## 总体流程

系统采用清晰的分层流程：

```text
扫描数据输入
  -> FootScanParser 解析为左右足矩阵
  -> HeatmapFeatureExtractor 提取足区特征
  -> FootRiskClassifier 生成非诊断性筛查说明
  -> Dashboard / PDF / API 输出结果
```

这种拆分的好处是解析、特征提取、规则筛查、可视化和接口输出彼此独立，后续接入真实硬件、替换模型或扩展报表时，不需要重写整个应用。

## 数据解析

当前演示协议是一个紧凑的二进制格式：

```text
byte 0: 宽度
byte 1: 高度
后续 width * height 字节: 左足矩阵
再后续 width * height 字节: 右足矩阵
```

每个压力点按 `0-255` 存储，解析后归一化为 `0.0-1.0`。解析器同时支持二进制、十六进制、二进制位串和 Base64 文本输入。

工程优势：

- 输入协议简单，便于压力垫、嵌入式设备或移动端快速对接。
- API 和 Razor 页面共用同一个解析器，减少前后端结果不一致。
- 对宽高、长度和空输入做边界检查，错误会转换为明确的用户提示。

限制：

- 当前协议没有包含设备编号、采样频率、标定参数或时间戳；生产硬件接入时建议扩展协议头或使用独立元数据。

## 热力图特征提取

`HeatmapFeatureExtractor` 会对左右足矩阵分别计算：

- 前足、中足、足跟平均负载
- 总负载和平均压力
- 峰值压力
- 有效接触面积比例
- 压力中心 `CenterX / CenterY`
- 热点数量
- 足弓指数 `ArchIndex`

区域划分采用行比例：

- 前足：高度前 35%
- 中足：高度 35%-70%
- 足跟：高度后 30%

热点使用归一化阈值 `0.88`，有效接触使用阈值 `0.18`。这些阈值集中在 `AnalysisOptions` 中，方便后续根据设备标定调整。

工程优势：

- 所有输出都来自实际矩阵计算，没有固定演示数值。
- 特征是可解释的基础统计量，便于排查传感器数据和前端展示问题。
- 左右足分别计算，再生成合并指标，适合做对称性比较。

限制：

- 当前特征是规则统计，不等同于经过临床验证的疾病判定模型。
- 不同压力设备的量程、分辨率和标定方式不同，阈值需要结合设备数据重新校准。

## 规则筛查分类

`FootRiskClassifier` 将特征转换为非诊断性说明，包括：

- 足弓类型倾向
- 步态负载模式提示
- 左右负载平衡
- 局部高压热点统计
- 足弓与接触面积分布
- 前足/足跟负载分布
- 左右负载与压力中心对称性
- 热力图识别质量

新增的结构化字段包括：

- `ForefootHeelRatio`：前足/足跟负载比例，限制在 `0.1-10.0`
- `ArchIndexCategory`：`High`、`Normal`、`Flat`
- `LoadSymmetryScore`：左右总负载对称性，范围 `0-1`
- `HotspotSeverity`：`None`、`Mild`、`Moderate`、`High`

工程优势：

- 规则透明，输出原因可以直接追溯到输入特征。
- 分类结果和说明文本分离，便于 API、Dashboard 和 PDF 重用。
- 报告内保留免责声明，避免把筛查提示误用为诊断结论。

限制：

- 这些规则适合演示和初筛，不应直接作为医疗判断依据。
- 更严谨的结论需要真实传感器校准、重复采集、临床样本和专业评估。

## WebGL / Canvas 热力图渲染

前端优先使用 WebGL2 渲染热力图，并保留 Canvas 降级路径。WebGL 渲染流程：

```text
归一化矩阵
  -> 8-bit 纹理
  -> Shader 双三次插值采样
  -> 足形 mask 裁剪
  -> 分级颜色映射
```

工程优势：

- 后端只需要输出紧凑 raw pressure 数据，不需要生成大图片。
- 渲染工作放在浏览器本地，适合动态热力图和实时刷新场景。
- WebGL 不可用时仍可通过 Canvas 路径显示，增强浏览器兼容性。

限制：

- 当前足形 mask 是演示用参数化形状，不是从真实足部轮廓分割得到。
- 颜色映射用于辅助观察压力分布，不代表绝对医学风险等级。

## DTW 传感器时间对齐

`DynamicTimeWarpingSensorAlignmentService` 使用动态时间规整对齐压力序列和加速度计序列。

算法步骤：

1. 从压力样本中提取 `Load` 数组。
2. 从加速度计样本中提取 `Magnitude` 数组。
3. 创建 `(n+1) x (m+1)` 代价矩阵，并将初始值设为无穷大。
4. 使用以下公式累计代价：

```text
cost[i,j] = abs(a[i] - b[j]) + min(
  cost[i-1,j],
  cost[i,j-1],
  cost[i-1,j-1]
)
```

5. 从矩阵右下角回溯，得到单调对齐路径。
6. 返回总成本、完整路径、轻量索引路径和归一化距离。

工程优势：

- 可以处理压力垫和手机加速度计不同采样频率的问题。
- 返回 `alignmentPath` 方便前端或第三方开发者检查对齐关系。
- `normalizedDistance` 便于比较不同长度序列的对齐质量。

限制：

- 当前代价函数只比较压力负载和加速度模长的绝对差。
- 对于长序列，完整 `O(n*m)` 矩阵会消耗更多内存；生产场景可考虑窗口约束或分段对齐。

## ONNX 步态分析接入

`OnnxGaitAnalysisService` 提供 ONNX Runtime 集成点。配置 `GaitAnalysis:ModelPath` 且模型文件存在时，服务会：

1. 将压力序列展平为 `float32[1, N]` 张量。
2. 使用配置的输入名运行 ONNX 模型。
3. 从输出概率中取 top-1 类别。
4. 映射到 `Normal`、`Pronation`、`Supination`、`Antalgic` 标签。

未配置模型时不会抛出异常，而是返回：

```json
{
  "label": "ModelNotConfigured",
  "confidence": 0,
  "isPlaceholder": true
}
```

工程优势：

- 应用可以先完成接口、权限、上传和展示流程，再接入真实模型。
- 模型文件不进入仓库，避免误导用户认为已有可用医学模型。
- 推理能力封装在独立类库中，便于替换 1D-CNN、Transformer 或其他序列模型。

限制：

- 仓库不包含训练数据和真实模型。
- 模型效果取决于训练数据、标签质量、设备一致性和验证流程。

## Dashboard 与 PDF 输出

Dashboard 后端直接返回 ECharts 可消费的配置：

- 左右受力平衡柱状图
- CoP 偏移散点图
- 前足/足跟负载堆叠柱状图
- 热点数量饼图

PDF 使用 QuestPDF 生成，包含：

- 报告标题和非诊断声明
- 指标表格
- 筛查分类
- 原始矩阵摘要
- 页脚和项目信息

工程优势：

- Dashboard、PDF 和 API 使用同一份特征计算结果，减少展示差异。
- ECharts 配置由后端生成，前端只负责渲染，便于其他客户端复用。
- PDF 输出适合留档、打印和演示，不依赖浏览器截图。

限制：

- 当前 PDF 是研究/教学格式，不是监管意义上的医疗报告。
- 报表样式和字段应根据真实业务场景继续调整。

## 开放 API 与多租户

项目提供硬件上传、分析、渲染帧、Dashboard、PDF、步态分析、传感器对齐和档案管理接口。租户 ID 可来自 Identity claims 或 `X-Tenant-Id`。

工程优势：

- 其他开发者可以将压力采集硬件通过 REST API 接入平台。
- 患者/运动员档案和硬件上传流程已经具备接口骨架。
- 当前内存存储适合演示，生产环境可替换为数据库实现。

限制：

- 当前档案存储为内存实现，重启后数据会丢失。
- 生产部署需要补充持久化、审计、权限策略、数据加密和合规审查。

## 适合继续扩展的方向

- 引入真实设备标定参数，按设备型号管理阈值。
- 增加动态步态序列采集格式，而不只依赖静态矩阵。
- 为 DTW 加入窗口约束、重采样策略和异常点处理。
- 接入经过验证的 ONNX 模型，并建立评估集和混淆矩阵。
- 将内存档案存储替换为数据库，并完善租户级权限策略。
- 增加端到端 UI 测试，覆盖上传、Dashboard、PDF 下载和 API 调用。
